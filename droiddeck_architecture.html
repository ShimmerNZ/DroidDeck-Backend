Step-by-step detail
1
Backend starts BottangoLiveDriver at startup, connecting outbound to Bottango on the Windows PC at host:59225 (configured in hardware_config.json → bottango.host). Auto-reconnects every 5s if not available.
main.py → bottango_live_driver.py _connect_loop()
2
On connect, sends BOOT\n → Bottango responds with hRQ,<code> handshake request. Driver replies btngoHSK,CUSTOM8,<code>,1 → Bottango goes green (live).
bottango_live_driver.py _handle_handshake()
3
Bottango sends rSVPin,<id>,<min_us>,<max_us>,<max_change>,<start_us> for each enabled effector. Driver registers an _Effector with the PWM range. When min > max, direction is inverted — preserved exactly as configured in Bottango.
Limits clamped independently against servo_config.json as a hardware safety backstop.
bottango_live_driver.py _handle_register_servo()
4
Bottango sends tSYN,<timeline_ms> time sync packets as you play or scrub. Driver records the Bottango timeline time and local wall clock at sync point to track elapsed time.
bottango_live_driver.py _handle_time_sync()
5
Bottango sends sC,<id>,<start_ms>,<dur_ms>,<p0>,<cpStartX>,<cpStartY>,<p3>,<cpEndX>,<cpEndY> bezier curve commands. Curves are queued per effector sorted by start time. Start time is clamped to now if already in the past (late network arrival).
For instant position commands (sCI), position is applied immediately and any queued curves are cleared.
bottango_live_driver.py _handle_set_curve() / _handle_instant_curve()
6
A 50Hz tick loop runs continuously. Each tick computes the current Bottango timeline time from the last tSYN + elapsed wall clock. For each registered effector, evaluates the active bezier curve using cubic bezier sampling (mirrors Bottango's BezierCurve.cpp). When a curve expires, the final endpoint position is dispatched so the servo always lands correctly.
bottango_live_driver.py _playback_tick() → get_pw_at()
7
Resolved PWM value (µs) passed to hardware_service.set_servo_position(channel_key, pw, "realtime"). Same path as joystick from this point — bypasses MotionMixer entirely.
bottango_live_driver.py _send_servo() → hardware_service.py
Channel mapping: Bottango effector ID 0–23 → m1_ch0–m1_ch23 (Maestro 1). ID 24–47 → m2_ch0–m2_ch23 (Maestro 2). Matches the export/import convention in bottango_converter.py.

Bypass note: Like the joystick path, the live driver goes directly to hardware_service — it does NOT pass through MotionMixer. No channel locking, constraint pipeline, or scene blending. Designed for isolated testing of individual servos from the Bottango timeline before exporting animations.
Architecture Notes
⚠ Three independent paths to Maestro: The Steam Deck frontend joystick (Path A) sends {type:"servo"} WebSocket messages directly to hardware_service.set_servo_position(). The Bottango live driver (Path C) also calls hardware_service directly from its 50Hz tick. Neither passes through the MotionMixer, ConstraintPipeline, or CommandDispatcher. Only the Bluetooth controller backend path goes through the full mixer. This means:

• The channel lock check in controller_input_handler.py only protects the Bluetooth controller path
• The Steam Deck frontend has no channel lock protection — it can send servo commands to scene-locked channels
• The Bottango live driver has no channel lock protection — intended for isolated testing only, not simultaneous use with scene playback
• During scene playback, if the Steam Deck joystick or Bottango live driver moves a locked channel, both the mixer (via batch) and the direct path send conflicting commands to the same servo
